# 営業企画のためのAI Business OS
> 読了時間: 3分

---

## 「AIはなんでもできる」は本当か？

ChatGPT や Claude に聞けばなんでも答えてくれる。そう感じている人は多いはずです。

でも実際に業務で使おうとすると、こんな問題に気づきます。

| 体験した問題 | AIの構造的な弱点 |
|---|---|
| 先週話したことを覚えていない | **記憶がない**（毎回ゼロから説明が必要） |
| 提案書に自信満々な嘘の数値が入っていた | **事実を作る**（ハルシネーション） |
| 同じ頼み方をしても毎回品質がバラつく | **一貫性がない** |
| 担当者によって提案書の質が全然違う | **個人のスキル依存** |
| 同じミスを何度でも繰り返す | **学習しない** |

これらは「プロンプトの書き方」の問題ではありません。**AIという技術の構造的な限界**です。

---

## Skillsを作っても解決できない理由

「じゃあ便利なスラッシュコマンドを作ればいい」という発想があります。でもSkillsは**単発の命令を実行するだけ**なので、構造的な問題は何も変わりません。

- 使い忘れたらゼロ
- 数値の誤りは依然として起きる
- チームで品質を揃えられない
- 失敗が蓄積されない

**構造的な問題は、構造で解決するしかない。**

---

## このテンプレートが解決する方法

役割を分けて、ルールをファイルに持ち、失敗を学習する仕組みを構造として組み込みます。

| 弱点 | このテンプレートの解決策 |
|---|---|
| 記憶がない | `memories/` にファイルとして永続化。顧客情報・過去の提案も覚えている |
| 事実を作る | 数値・競合情報に「要確認」フラグを自動付与。人間が承認するまで完了にならない |
| 一貫性がない | `.clauderules` に営業企画の判断基準をファイルで固定 |
| 個人のスキル依存 | リポジトリをForkするだけでチーム全員が同じルールで動く |
| 学習しない | 失敗パターンを蓄積し、次回から自動的に回避する |

## 営業企画の現場でいうと

```
「A社向けの提案書を作って。
　業種は中規模小売、課題は在庫管理の非効率化」
　→ Executive Summary・課題→解決策→効果の構成で出てくる
　→ 数値には「要確認」フラグが付く（そのまま使えない仕組み）
　→ 次回は「A社の前回提案を踏まえて」と言えば覚えている

「来週のB社商談の準備をして」
　→ 想定QA・提案ポイント・トーク構成が出てくる
　→ 「前回この切り返しがうまくいかなかった」という学習が活きる
```

**普通にClaudeに聞くとの違い：覚えていて、数値の誤りを構造的に防ぎ、学習する。**

## 頼める作業の例

| 作業 | 具体的な頼み方 |
|---|---|
| 提案書作成 | 「〇〇業界の中規模企業向け提案書。課題は△△。競合との差別化も含めて」 |
| 商談準備 | 「明日の商談の想定QAと切り返しトークを作って。相手は慎重な決裁者タイプ」 |
| 競合比較 | 「自社とA社・B社の比較表を作って。顧客にとっての価値で表現して」 |
| 議事録まとめ | 「今日の商談内容をまとめて。決定事項・懸念点・次のアクションを分けて」 |
| 顧客分析 | 「この業界の顧客が抱えやすい課題と、刺さる提案ポイントをまとめて」 |
| 営業資料の改善 | 「この提案書のWhyが弱い。顧客視点で書き直して」 |

## 使い始め方

**所要時間: 30分以内**

1. GitHubで「Fork」ボタンを押す
2. Claude Code（VSCode拡張 or `claude.ai/code`）でフォルダを開く
3. 「営業企画として使いたい。どのフォルダを削除すればいい？」と聞く
4. 指示通りに削除して完了

あとは日本語で話しかけるだけです。

---

## よくある疑問

**Q. 顧客名や社外秘の情報を入力して大丈夫？**
A. 実名・社外秘データはそのまま入力しないでください。「A社」「B社」のように仮名化してから渡してください。

**Q. 出てきた数値をそのまま提案書に使っていい？**
A. 使わないでください。数値・競合情報は必ず自分でファクトチェックしてから使用してください。テンプレートは「要確認」フラグで区別する設計になっています。
